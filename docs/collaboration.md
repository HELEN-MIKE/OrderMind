# OrderMind 项目分工说明

## 1. 分工原则

本项目采用“业务验收 + 技术实现”分工。

你不需要写代码，也不需要处理技术细节。你的主要角色是客户代表、业务规则提供者和验收负责人。我负责代码实现、测试、文档、运行验证和交付整理。

## 2. 你的职责

### 2.1 提供真实样例

请尽量收集客户真实订单文件：

- Excel 订单。
- Word 订单。
- PDF 订单。
- TXT 或邮件正文订单。
- 格式异常或容易出错的订单。

第一阶段建议至少准备：

```text
Excel 订单：5-10 份
Word 订单：3-5 份
PDF 订单：3-5 份
异常订单：3 份
```

这些样例决定解析器和校验规则是否真的贴近客户现场。

### 2.2 确认业务字段

你负责判断哪些字段是客户必须要的。

当前候选字段包括：

- 货号。
- 品名。
- 数量。
- 单价。
- 小计金额。
- 总金额。
- 材质。
- 交货期。
- 付款方式。
- 包装要求。
- 单位。
- 检测要求。
- 出口规范或合规条款。

你只需要用业务语言确认字段重要性，不需要设计数据库或写代码。

### 2.3 提供校验规则

你可以用自然语言描述规则，例如：

```text
货号必须是两个大写字母 + 横杠 + 四位数字。
总金额允许 0.05 以内尾差。
单价最多保留两位小数。
数量必须大于 0。
付款方式必须包含 T/T 或 L/C。
包装要求必须出现“外箱”或“托盘”。
```

我会把这些业务规则转换成模板配置和校验逻辑。

### 2.4 做验收反馈

你需要打开应用，按真实业务流程试用，并反馈：

- 哪些字段识别错了。
- 哪些校验提示不符合业务。
- 哪些界面文案客户看不懂。
- 哪些操作步骤不顺。
- 哪些问题是首版必须解决，哪些可以后续版本处理。

## 3. 我的职责

### 3.1 技术实现

我负责：

- 项目结构设计。
- 订单解析器实现。
- 校验规则引擎实现。
- 本地 Web 界面实现。
- 中英文界面实现。
- 登录、密码修改、本地用户存储实现。
- 首次使用指引实现。
- 后续桌面应用打包方案设计。

### 3.2 文档输出

我负责维护：

- 原始需求整理。
- 实现路线。
- 分工说明。
- 桌面应用规划。
- 运行说明。
- 测试说明。
- 第一版已支持和未支持清单。

### 3.3 测试与验证

我负责：

- 编写自动化测试。
- 运行测试。
- 验证样例订单。
- 检查语法和模块可编译性。
- 记录当前限制和后续风险。

### 3.4 交付整理

我负责把项目整理到指定路径：

```text

```

## 4. 推荐协作节奏

### 阶段一：确认方向

你确认：

- 产品名 OrderMind 订单智脑。
- 本地桌面应用方向。
- 中文和英文双语。
- 本地账号登录。
- 首次登录修改密码。
- 首次使用操作指引。

### 阶段二：验证第一版

你使用样例文件跑通流程：

```text
登录 -> 修改密码 -> 上传订单 -> 选择模板 -> 开始审单 -> 查看结果
```

### 阶段三：真实样例验收

你提供真实订单，我负责逐步增强解析和规则。

### 阶段四：客户演示

我整理演示说明、已支持范围、后续路线和风险边界，你负责向客户确认是否符合业务预期。

## 5. 决策边界

你负责决定：

- 客户最关心哪些文件格式。
- 哪些字段必须首版支持。
- 哪些校验规则必须首版支持。
- 界面中英文文案是否符合客户习惯。
- 识别准确率是否达到可演示标准。

我负责决定：

- 技术架构。
- 模块拆分。
- 测试策略。
- 代码规范。
- 安全实现方式。
- 后续可维护性方案。

如果技术实现会影响业务效果，我会先说明取舍，再继续实现。
