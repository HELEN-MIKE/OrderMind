# OrderMind 桌面应用规划

## 1. 桌面应用目标

OrderMind 最终应交付为可双击运行的桌面应用，面向非技术用户。

目标体验：

```text
打开应用
  -> 登录
  -> 首次登录修改密码
  -> 查看首次使用指引
  -> 上传订单
  -> 选择规则模板
  -> 开始审单
  -> 查看中英文结果
  -> 导出报告
```

## 2. 推荐桌面技术路线

推荐优先使用：

```text
Tauri + React/Vue + 本地核心服务
```

原因：

- 应用体积较小。
- 安全边界较清晰。
- 前端界面能力强。
- 可调用本地文件系统和本地 worker。
- 适合离线工具。

备选路线：

```text
Electron + React/Vue + Node/Python worker
```

优点是生态成熟、开发快；缺点是应用体积更大。

## 3. 当前第一版形态

当前第一版使用：

```text
Python 标准库 Web 服务 + HTML 页面
```

这样做是为了先验证业务闭环：

- 不依赖第三方包。
- 离线可运行。
- 可以快速演示登录、上传、校验和双语。
- 后续桌面壳可以复用核心解析、校验、认证和 i18n 资源。

## 4. 双语设计

第一版已有 `ordermind/i18n.py`。

原则：

- 展示文案使用语言 key。
- 业务校验 code 使用稳定英文，不随语言变化。
- 中文和英文 UI 都必须覆盖登录、改密、首次使用指引、上传、审单结果。
- 后续报告导出可以按用户语言输出。

## 5. 登录与密码策略

第一版支持本地账号密码登录。

当前策略：

- 默认账号：`admin`。
- 默认密码：`Admin123456`。
- 首次登录必须修改密码。
- 密码至少 8 位，并包含字母和数字。
- 密码使用 PBKDF2 加盐哈希保存，不保存明文。

后续桌面版建议：

- 使用 SQLite 保存用户。
- 增加用户管理界面。
- 增加角色权限：管理员、审单员、只读查看。
- 增加操作日志。
- 支持密码重置。

## 6. 首次使用指引

第一版首页已包含四步指引：

```text
第 1 步：选择订单文件
第 2 步：选择校验模板
第 3 步：点击开始审单
第 4 步：查看结果并确认
```

桌面版建议升级为：

- 首次启动弹出引导。
- 每一步高亮对应控件。
- 用户可点击“不再显示”。
- 设置页可重新打开新手引导。
- 中英文文案同步。

## 7. 本地数据规划

第一版使用 JSON 文件：

```text
data/users.json
templates/*.json
```

桌面产品化后建议迁移到 SQLite：

```text
users
templates
orders
order_lines
validation_reports
audit_logs
settings
```

## 8. 打包规划

项目规则：每个发给客户试用的版本都必须产出 Mac 和 Windows 安装包，并提供升级说明。不能只交付源码或本地运行脚本。

### 阶段一：本地开发版

```bash
python3 run_app.py
```

### 阶段二：Python 可执行文件

使用 PyInstaller 打包本地服务：

```bash
pyinstaller --onefile run_app.py
```

### 阶段三：正式桌面壳

使用 Tauri 或 Electron：

- 前端负责界面。
- 本地 worker 负责解析和校验。
- SQLite 保存账号、模板和记录。
- 文件处理全程在本地完成。

推荐正式产物：

```text
Mac:
  OrderMind_版本号_aarch64.dmg
  OrderMind_版本号_x64.dmg

Windows:
  OrderMind_版本号_x64-setup.exe
  OrderMind_版本号_x64.msi
```

## 8.1 升级规划

推荐使用 Tauri updater 实现应用内升级。

升级流程：

```text
应用启动
  -> 检查更新清单
  -> 发现新版本
  -> 显示更新说明
  -> 下载签名安装包
  -> 校验签名
  -> 安装并重启
```

试用阶段可以先使用手动升级：

- 应用提示新版本。
- 客户下载安装包。
- 覆盖安装。
- 本地用户、模板和历史数据保留。

正式阶段再升级为自动更新。

## 9. 安全边界

第一版和桌面版都应遵守：

- 不把订单文件上传到公网。
- 不保存明文密码。
- 不在日志中输出完整订单敏感内容。
- 解析失败时给出可理解错误信息。
- 低置信度字段必须提示人工确认。

## 10. 后续验收重点

客户验收时应重点看：

- 非技术用户能否按指引完成上传和审单。
- 中文/英文界面是否完整。
- 登录和首次改密是否清楚。
- 真实订单字段识别是否准确。
- 校验结果是否符合业务规则。
- 报告是否能支持客户内部流转。
