# OrderMind 订单智脑：需求整理与实现路线

## 1. 客户原始需求

客户希望建设一个订单上传与智能校验工具，支持 `.doc`、`.docx`、`.xls`、`.xlsx`、`.pdf`、`.txt` 格式订单上传。

核心诉求包括：

- AI 自动识别表格结构，兼容合并单元格和复杂排版。
- 精准提取货号、品名、数量、单价、小计金额、总金额、材质、交货期、付款方式、包装要求等核心字段。
- 无需固定模板排版。
- 可视化配置校验规则，支持多套模板保存和一键切换。
- 校验规则包括数量类、金额类、格式类、必填项和合规类。
- 希望支持完全离线使用。
- 最终桌面应用需要支持中文和英文两种语言。
- 桌面应用需要有登录界面，支持不同用户通过账号和密码登录。
- 支持用户首次登录后修改密码。
- 首次打开应用后，应提供清晰的点击操作指引和配文说明。
- 每个发给客户试用的新版本都应打包成可安装的 Mac 和 Windows 应用。
- 新版本发布后，应支持客户升级，并尽量保留本地账号、模板和审单数据。

## 2. 产品命名

产品名确定为：

```text
OrderMind 订单智脑
```

产品定位：

```text
离线智能审单与订单校验工具
```

中文口号：

```text
读懂订单，识别风险。
```

英文口号：

```text
Understand Orders. Detect Risks.
```

## 3. 核心判断

客户希望“复杂 AI 文档解析”和“完全离线”同时成立。这个方向可做，但不能承诺第一版就万能识别所有复杂订单。

更可靠的产品路线是：

```text
本地桌面应用
+ 可插拔文档解析引擎
+ AI 辅助字段识别
+ 确定性规则校验
+ 人工确认兜底
+ 模板沉淀复用
```

AI 不应该直接给最终结论。推荐分工是：

- AI/解析器负责猜：哪个表是订单明细，哪列像货号，哪段像付款方式。
- 规则引擎负责判：数量是否加总正确，单价乘数量是否等于小计，总金额是否一致，货号格式是否合规。
- 用户负责确认：低置信度字段、复杂格式和业务例外。

## 4. 推荐架构

```text
桌面应用壳：Tauri 或 Electron
前端界面：React 或 Vue
本地核心服务：Python / Rust / Node 均可，第一版使用 Python 标准库实现
文档智能引擎：后续接入 Docling、PaddleOCR、openpyxl、python-docx、PDF 解析与 OCR
本地数据：SQLite 或 JSON，第一版使用 JSON 文件
本地 AI：后续可选 Ollama / llama.cpp
```

统一数据流：

```text
文件导入
  -> 格式转换与解析
  -> 表格结构识别
  -> 统一订单数据模型
  -> 字段候选与证据
  -> 人工确认
  -> 规则模板校验
  -> 报告导出
```

## 5. 第一版 MVP 范围

第一版目标是形成可运行闭环：

- 本地 Web 工作台。
- 中文/英文界面切换。
- 登录页。
- 默认本地管理员账号。
- 首次登录强制修改密码。
- 首次使用四步操作指引。
- TXT、CSV、TSV、XLSX、XLSM 解析。
- 字段抽取：货号、品名、数量、单价、小计、材质、单位、总金额、总数量、付款方式、包装要求、交货期。
- 默认规则模板。
- 金额、数量、格式、单位、必填项校验。
- JSON/CSV 报告输出。
- 自动化测试覆盖核心解析、校验、模板、双语、认证。

暂不承诺：

- `.doc` 原生解析。
- `.docx` 复杂表格解析。
- 复杂 PDF 表格恢复。
- 扫描 PDF OCR。
- 本地大模型语义理解。
- 企业级角色权限和审计。
- 云端同步或多人协同。

## 6. 后续版本路线

### 第二版：文档格式增强

- 接入 openpyxl，增强 Excel 合并单元格、公式、多个 sheet 和样式识别。
- 接入 python-docx，支持 DOCX 表格与段落解析。
- 使用 LibreOffice 将 DOC 转换为 DOCX 后解析。

### 第三版：PDF 与 OCR

- 接入 pdfplumber / PyMuPDF 处理电子 PDF。
- 接入 PaddleOCR / PP-Structure 处理扫描 PDF、图片订单和复杂表格。
- 增加坐标证据，让用户能定位字段来源。

### 第四版：AI 语义增强

- 接入本地 LLM，用于同义表达匹配和条款识别。
- 针对付款方式、包装要求、材质标准、检测要求、出口规范建立可配置规则库。
- 对低置信度字段提示人工确认。

### 第五版：桌面应用产品化

- 使用 Tauri 或 Electron 打包为桌面应用。
- 使用 SQLite 管理用户、模板、历史订单和审单记录。
- 支持导出 Excel/PDF 审单报告。
- 支持客户模板管理、版本冻结、操作日志和权限角色。
- 每次发布产出 Mac `.dmg` 和 Windows `.exe/.msi` 安装包。
- 接入应用内升级或试用阶段手动升级提醒。
- 建立版本号、变更日志、升级清单和数据迁移机制。

## 7. 质量原则

- 代码必须有清晰注释，解释模块职责、关键算法和边界限制。
- 金额和数量必须使用 Decimal，不能用 float。
- 密码不得明文保存，必须使用加盐哈希。
- 校验结果必须有稳定 code，便于中英文展示和后续统计。
- 自动识别结果应保留来源证据和置信度。
- 复杂或低置信度结果必须允许人工确认。
- 客户试用版本必须有安装包、版本号、变更日志和升级说明。
